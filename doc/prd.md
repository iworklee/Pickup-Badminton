一个支持 **“中途加入”的羽毛球多人转（轮转打路人局/俱乐部活动）Web页面，核心难点在于排阵算法的动态化**。

传统的排阵（如你截图中底部的“生成说明”）是**静态排阵**：一开始有几个人，就按照固定的数学矩阵排好所有的场次。一旦有人中途加入，传统的静态排阵就会彻底崩溃。

为了支持“中途加入”，我们需要将设计思路从“生成固定赛程”转变为**“动态队列调度（排队叫号机制）”**。

以下是完整的Web端设计方案，分为**核心逻辑、页面UI设计、以及技术实现建议**：

---

### 一、 核心逻辑设计：动态权重排阵算法

要支持中途加入，系统必须拥有一个实时的“计算大脑”。每次有一场比赛结束，系统实时计算下一场谁上。

**计算权重规则（得分越高越先上场）：**

1. **出场次数：** 出场次数最少的人，权重最高（中途加入的人出场次数为0，会立刻获得最高优先级被安排上场）。
2. **休息时长：** 连续休息场次越多/等待时间越长的人，权重越高。
3. **搭档与对手重复率：** 尽量避免连续两场和同一个人搭档，尽量避免连续两场打同一个对手（降低权重）。
4. **性别平衡（可选）：** 根据配置的让分规则，系统尽量凑成男双vs男双，混双vs混双。如果凑不齐，再应用让分规则。

---

### 二、 页面UI交互设计 (Mobile First 移动端优先)

整个Web应用可以分为四个核心页面/模块：

#### 1. 创建比赛页 (参考你的截图并优化)

- **基础信息：** 标题、日期、场地数量（**新增**：因为动态排阵需要知道有几个场地同时打）。
- **初始参赛人员：** 姓名、性别。支持微信群接龙文本一键粘贴导入。
- **让分设置：** 保留截图中的让分规则（男双让混双等）。
- **模式选择（新增核心）：**
  - *模式A：固定场次制*（原始模式，不支持中途加入）。
  - *模式B：动态轮转制*（**推荐**，支持中途加入，按时间或大家打累了随时结束）。
- **底部按钮：** “开始活动” (代替“生成比赛”)。

#### 2. 实时看板页 (Live Dashboard - 核心工作台)

这是活动开始后的主页面，所有人可以通过分享的链接查看。

- **顶部：进行中的比赛 (正在打)**
  - 卡片式展示：`场地1：张三(男)/李四(女) VS 王五(男)/赵六(男)`
  - 显示本场让分提示：`王/赵 需让 3分`
  - 操作按钮：【录入比分】、【结束本局】。
- **中部：候场队列 (即将上场)**
  - 显示系统动态计算出的下一场名单（Next Up）。
  - 让大家有准备去热身。
- **悬浮按钮 (FAB - 核心操作)：**
  - 右下角放置一个醒目的 `**+ 添加选手 (中途加入)`** 按钮。点击后弹窗输入姓名性别，该选手会立刻被加入系统的动态计算池，并在下一场被优先安排。
  - `**- 选手早退**` 按钮：如果有人提前走，可以将其状态设为“休息/离场”，系统不再安排他。

#### 3. 录入比分弹窗

- 极简的数字输入框。
- 如果之前设置了让分，系统自动在初始比分上加上让分（例如 0 : 3 开始）。
- 录入完成后，系统自动更新排行榜，并**触发下一次动态排阵计算**。

#### 4. 数据统计与排行榜页 (Leaderboard)

- **个人战绩表：** 胜率、净胜分、出场次数。
- **搭档胜率榜：** 哪两个人搭档胜率最高。
- 因为是动态轮转，每个人的出场次数可能不同，所以排行榜应优先以**胜率**或**场均净胜分**来排序，而不是总积分。

---

### 三、 针对“中途加入”的特殊场景处理

1. **新玩家加入时机：** 新玩家点击加入后，系统提示“已加入候补名单”。当当前任意一块场地的比赛结束时，系统进行下一轮排阵，新玩家因为出场次数为0，会被100%选入下一场比赛。
2. **费用分摊（可选高级功能）：** 中途加入的人，打的场次少，如果最后要AA场地费，系统可以根据“实际出场次数”或“在场时长”给出AA建议金额。
3. **强制干预：** 虽然有算法，但必须提供**手动换人**功能。如果系统排了某人，但他刚好去洗手间了，管理员可以长按该选手，选择“换人”，系统重新从队列中拉取下一位。

---

### 四、 技术实现方案建议 (Web前端+后端)

- **前端框架：** React 或 Vue.js。适合做这种状态频繁更新的单页应用(SPA)。
- **UI组件库：** Vant (Vue) 或 Ant Design Mobile (React)，非常适合做类似微信小程序风格的移动端Web。
- **状态同步 (关键)：** 强烈建议使用 **WebSocket** 或 Server-Sent Events (SSE)。
  - *原因：* 这是一个多人参与的活动。组织者在手机上点击“比赛结束”或“添加中途加入人员”后，其他坐在场边看手机的球友的页面应该**实时自动刷新**候场名单，而不需要手动刷新浏览器。
- **本地缓存：** 使用 `localStorage`。防止组织者不小心关掉浏览器网页导致比赛数据丢失。每次操作都存入本地，重新打开网页时恢复状态。

### 总结设计亮点：

相比于传统打法，这个设计的最大突破在于引入了**“动态队列”**和**“实时状态机”**。它不仅完美解决了中途加人的问题，还顺带解决了有人临时有事早退、或者某人连续打了太累需要强制休息一轮的现实痛点，非常贴合真实的羽毛球俱乐部活动场景。
